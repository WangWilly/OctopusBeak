# 資料問題漸進式展開流程

## 目標

讓使用者從回報內容直接開始排除錯誤匯入，並在同一張卡片內逐步完成來源確認與影響預覽，避免頁面切換造成脈絡中斷。

## 介面

- 初始卡片移除「調查中」狀態標籤。
- 初始主按鈕由「開始診斷」改為「排除錯誤匯入」。
- 移除診斷／預覽畫面上方的 breadcrumb 與錯誤紀錄 disclosure。
- 錯誤資料仍保留在既有狀態模型中；本次只移除該頁的錯誤紀錄展示元件。
- 點擊主按鈕後，原卡片原地展開步驟 1「回報內容」摘要與步驟 2「確認來源」。
- 選擇來源並預覽後，步驟 2 收合為摘要，步驟 3「預覽影響」在同一卡片下方展開。
- 保留現有來源資料、原始資料列、影響數值、停用原因、確認勾選與操作按鈕。

## 動畫

- 使用 Svelte 內建 `slide` transition 處理新增內容的高度與透明度變化，不新增依賴。
- 動畫只套用到新展開的階段內容；已完成摘要列維持穩定，避免整頁跳動。
- 遵守 `prefers-reduced-motion`：使用者要求減少動態時將 transition duration 設為 0。

## 狀態與資料流

- 沿用現有 `list → diagnosis → preview` 狀態與事件，不新增狀態模型或抽象層。
- `open-diagnosis` 觸發第一段展開；`preview` 觸發第二段展開。
- 後續停用、成功、稽核與還原流程不變。

## 驗證

- UI source check 驗證新文案、Svelte transition 與移除的元件。
- Electron/CDP 驗證點擊前後仍是同一路由與同一卡片，下一階段有 transition，主要操作可完成。
- 驗證窄視窗無水平溢出、鍵盤與 reduced-motion 基本可用性、console 無錯誤。

## 非目標

- 不改變錯誤紀錄的保存方式。
- 不接上真實 ledger 寫入。
- 不重做成功、稽核或還原畫面。
